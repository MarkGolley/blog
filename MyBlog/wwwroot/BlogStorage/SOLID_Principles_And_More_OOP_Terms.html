SOLID Principles

These are the fundamental principles that help keep object-oriented code readable, maintainable, and far less painful to change.

  • S – Single Responsibility: Each class should have one responsibility. For instance, one class to generate a file and another to save it, rather than stuffing everything into one large class.
  • O – Open/Closed: Classes should be open to extension but closed to modification. I saw an example with shapes: instead of altering a method each time, you define an interface and add new shapes without changing existing code.
  • L – Liskov Substitution: (I still stumble over the name.) Derived classes should behave in ways the base class expects. You should be able to substitute a derived class for its base without breaking anything. This is one reason people favour interfaces over inheritance when behaviour varies.
  • I – Interface Segregation: It is better to have several small, focused interfaces than a single large one. If you find yourself writing throw new NotImplementedException(), the interface is probably too broad.
  • D – Dependency Inversion: High-level classes should not depend on low-level details. Inject abstractions rather than instantiating concrete types directly. This makes the code easier to test and swap later.

Class naming

I used to call a subclass a "child class", but the more standard terms are:

  • Superclass / Base Class: A class that is inherited from.
  • Subclass / Derived Class: A class that inherits from the base.

Other terminology

I came across the term destructor (as opposed to a constructor). It exists in C#, though it is rarely used. That led me down a terminology rabbit hole:

  • Abstract Class: Cannot be instantiated. May contain abstract methods (which must be implemented) as well as shared methods.
  • Static Class: Cannot be instantiated. Contains only static members, usually helpers.
  • Sealed Class: Cannot be inherited from.
  • Polymorphism: Code can operate with different implementations of a type via an interface (e.g. ILogger).
  • Encapsulation: Keeping internal state private and only exposing what is needed. Something we could improve in the Warp Engine.
  • Members of a Class: All the class "bits and pieces" – fields, properties, methods, constructors, etc.
  • Composition: One class owns another and creates it directly. Tightly coupled.
  • Aggregation: Similar to composition but looser. The dependent class can exist independently (often passed in).
  • Association: A general term for when classes are related or aware of one another in some way.
