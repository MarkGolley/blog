The 16-Hour AI Moderation Build

I thought this would be a quick feature: add AI moderation, add an admin page, done. In reality it became a 16-hour build touching authentication, security headers, anti-forgery behaviour, UI feedback, test coverage, and deployment quirks.

It was definitely worth doing though. The comment system now behaves much more like a real production feature rather than a basic form that accepts everything.

What I was trying to solve

I wanted two clear outcomes:

  • New comments and replies should be checked before they appear publicly
  • I need an admin flow to approve or delete comments quickly

The tricky part is that this is not one feature, it is a chain of features that all need to work together without breaking the reader experience.

Part 1: AI moderation in the comment pipeline

I added an AIModerationService that calls OpenAI's moderation endpoint using omni-moderation-latest. Every comment is checked in CommentService.AddCommentAsync before it is stored.

The coolest part is this: moderation now runs automatically on every comment, all the time. Instead of manually reading everything first, AI does the first pass instantly and I only step in for edge cases. I also added spend limits (currently $1) so I can keep experimenting safely.

Beyond this, an important design choice was fail-safe behaviour:

  • If the API key is missing, the comment is held for manual review
  • If the moderation call fails, my budget is exceeded, or the payload is unexpected, the comment is held for manual review
  • Only clearly safe comments are auto-approved

That sounds strict, but strict is the point for moderation. I would rather review one extra comment than accidentally publish a harmful one.

Part 2: separating pending vs visible comments

Each comment now has an IsApproved flag. Public blog pages only render approved comments, while pending comments are visible in admin.

On submit, I also added clear user feedback. If a comment is held back, the user is redirected to the post with a moderation status and sees a clear banner message near the form. This avoids the "did my comment just disappear?" confusion. Replies go through the same moderation path, and the existing thread depth limits still apply.

Part 3: building an admin layer that is simple but usable

I added an AdminController with a cookie-auth login flow and a dashboard listing pending comments. Credentials are read from environment variables. Once logged in, I can:

  • Approve a pending comment
  • Delete a comment
  • Delete full comment threads (parent and all replies)

That last point mattered. Deleting just the parent and leaving orphaned replies would create messy data and confusing UI, so delete operations now walk the thread and remove descendants too.

Optional deep dive Cookie auth flow explained. Skip this section if you already know cookie-based auth.

Step 1: login request

You post username/password to /Admin/Login. The server checks those values against admin credentials from environment variables.

Step 2: session cookie is created

If credentials match, the server creates a ClaimsIdentity with an Admin role and calls SignInAsync. ASP.NET Core then issues an encrypted auth cookie called __session.

Step 3: protected routes trust the cookie

On later requests, the browser automatically sends that cookie. Any action marked with [Authorize] (for example approve/delete actions) only runs if that cookie is valid.

Step 4: logout clears access

When logout runs, SignOutAsync clears the auth session, so protected admin actions are blocked again until you log back in.

I also tightened the login setup so it is safer in day-to-day use, not just in local testing.

In simple terms: the login cookie is harder to steal, it only travels on secure connections, and admin pages are always fetched fresh so people do not see stale logged-in pages from cache.

This is still a simple admin model (shared credentials from environment variables), but it is a practical step up from no auth at all and it keeps moderation tooling private.

The awkward bits that consumed most of the time

The big effort was not writing the happy path. It was handling everything around it. Every single issue below was catching me out at one point or another:

  • Anti-forgery token issues on repeated form interactions: if someone leaves a page open for a while or retries quickly, that safety token can fail and block a valid submit.
  • Recovering from 400 responses with a form=expired hint: instead of a dead-end error page, the user is sent back to the post with a clear prompt to refresh and try again.
  • Honeypot handling with clearer messaging: hidden fields are used to catch bots, but some browser tools auto-fill them, so I added friendlier feedback when that happens.
  • No-store caching on dynamic/admin pages: login and moderation pages are always fresh, so users are not looking at old state after login, logout, approve, or delete actions.
  • Case-insensitive admin login handling: small input differences like uppercase letters no longer cause unnecessary login failures.

Each one looks minor on its own, but together they are the difference between "works on localhost" and "works reliably". The volume of time being spent crushing these bugs made we look deeper into testing...

Testing was what made this feel safe to ship

Given the raft of challenges faced, the growing code base and finding errors in production that didn't occur locally meant it became valuable to increase the scope and type of testing, a small upfront spend to get more test coverage would save me in the long run. I ended up with a mix of new tests around the flow:

  • Unit tests for moderation service behaviour (safe, flagged, missing key, unexpected payload)
  • Integration tests for comment visibility, moderation redirects, admin actions, and auth flows
  • Live moderation verification tests (when API key is available)
  • Playwright E2E checks for mobile moderation banners and login/logout journeys

Without these tests I would not trust changes around security and moderation because regressions here are easy to miss and expensive when they leak into production.

What I learned

A few things stood out:

  • Moderation is a reliability feature, not just an AI feature
  • Fail-safe defaults are essential when external APIs are involved
  • Admin tooling needs just as much thought as public UX
  • A lot of engineering time can go into edge cases and operational behaviour
  • Automated tests are the only way to keep confidence while iterating quickly

Overall, this was one of those features that looked small and ended up teaching me a lot. The biggest win is that AI now handles first-pass moderation 24/7, while I keep control of the final quality bar through admin review and tests.
