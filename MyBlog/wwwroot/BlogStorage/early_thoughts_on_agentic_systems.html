Experimenting with Agentic AI in .NET (and Realising It’s Not Just Chat Anymore)

I use AI tools a lot. Code completion, refactoring suggestions, explaining errors, generating test cases. At this point it feels normal and really almost expected. Because of that, I thought I had a pretty solid grasp of where AI in engineering was heading.

Recently though, I was introduced to something slightly different. Not just “AI that suggests things”, but AI that can decide which tools to use, call them, gather data, and then synthesise a response. This started to feel different! It wasn’t just chat anymore it was orchestration.

From assistant to orchestrator

Most AI usage today is some text in and some text out. You ask a question and it replies. You paste code in and it improves it (sometimes?! Most of the time?!). It's really helpful but quite contained. What I’ve been experimenting with is closer to what people are calling “agentic systems”. Instead of only generating text, the model can:

  • Interpret a request
  • Decide which registered function to call
  • Execute that function
  • Observe the result
  • Call another function if needed
  • Then synthesise a grounded answer

This “reason -> act -> observe” loop is supported by frameworks such as LangChain and Microsoft Semantic Kernel. I focused on Semantic Kernel because I work in .NET and it integrates cleanly with C#.

The important part is not the framework itself, it’s the pattern. The model becomes a reasoning layer sitting on top of real tools and structured data which feels like a different abstraction to what I am used to.

Building a tiny demo to understand it properly

Instead of just reading about it, I built a small .NET console application to see how it actually behaves.

The demo is intentionally simple. It registers two tools:

  • GetOrderById(int orderId) – reads a fictional order from a local JSON file
  • SearchOrderLogs(int orderId) – searches a local log file for related entries

When I type:

Investigate order 123 and explain what went wrong.

The model does not just answer generically. It:

  • Calls GetOrderById
  • Calls SearchOrderLogs
  • Combines the results
  • Returns a grounded explanation

The full minimal example is available on my GitHub which demonstrates the core pattern clearly.

What felt powerful

The most interesting part wasn’t that it worked, it was how it worked. Instead of manually opening files, grepping logs, and piecing together context, the model could look in multiple places at once and synthesise a summary. It reduces the distance between a question and an understanding.

This starts to resemble a kind of natural language control layer over structured systems. We're now able to add a reasoning layer that sits above APIs and services and that feels significant.

What felt fragile

It’s also very obvious that this is not magic. If a data file is missing or if the tool description is unclear, the model can become confidently wrong. It depends heavily on:

  • Clear function schemas
  • Stable environment setup
  • Predictable outputs
  • Defensive error handling

At one point I had file paths wrong and the system “worked”, but returned misleading conclusions because the tools were returning empty results. The model happily summarised incomplete information which in itself was a useful lesson.

Prompt engineering quickly turns into interface engineering. The reliability of the system depends far more on how well the tools are designed than on how clever the prompt is and the more we put into this design the better the results will no doubt be.

What I learned about function schemas

One concept I did not properly understand at first was what people mean by “function schemas”. When you register a C# method as a callable tool, the model does not see your source code. It sees a structured contract describing the function. Under the hood, the model receives something closer to a JSON definition of the function:

{
  "name": "GetOrderById",
  "description": "Get an order by its numeric ID.",
  "parameters": {
    "type": "object",
    "properties": {
      "orderId": {
        "type": "integer",
        "description": "The numeric ID of the order"
      }
    },
    "required": ["orderId"]
  }
}

That structure is the schema. It defines what the function is called, what it does, what parameters it expects, and what types those parameters should be. If the schema is vague or unclear, the model can call the function incorrectly or chose the wrong tool.

Capturing process as callable capability

Another thing I noticed is how this pattern naturally encodes process. When you define tools, you are formalising workflows:

  • What to check first
  • Which data sources matter
  • How information should be summarised

Some of the knowledge that would normally live in documentation becomes executable. Instead of reading a runbook and manually following steps, the system can perform those steps and return structured output. This doesn’t remove the need for engineering judgement but it does help for repeatable tasks and it feels like a small step towards capturing your knowledge in code.

The slightly uncomfortable bit

I enjoy (mostly) investigating issues - there is something satisfying about following your nose (and learning how to do it better) through logs and slowly narrowing down the root cause so if an AI system performs the first layer of investigation, does that remove some of that satisfaction?

For me, maybe.

But, on the other hand automating the repetitive parts feels sensible and it frees up our attention for more interesting problems. I don’t have a strong conclusion yet as I’m still very much experimenting but the shift from “AI as assistant” to “AI as orchestrator” feels real.

Why I think this matters all this matters

I don’t think this means full autonomy is around the corner becase these systems are bounded by the tools we define and the guardrails we build. But architecturally, it introduces a whole new layer - a reasoning layer coordinating capabilities

Whether this becomes reliable and trusted will depend less on raw model capability and more on how deliberately we design around it. This experiment has mostly focused on the simplest possible orchestration pattern: one model, a couple of tools, and a clear loop.

There are obvious next steps.

  • Designing multiple specialised agents and allowing them to hand tasks between each other (chaining)
  • Introducing explicit human-in-the-loop approvals for state-changing operations
  • Exploring retrieval-augmented generation (RAG) in more depth
  • Understanding emerging standards like the Model Context Protocol (MCP)

For now, I’m treating this as an early exploration rather than a conclusion. I’ll likely write follow-up posts as I experiment further and understand the trade-offs more clearly. If nothing else, it feels like a space worth paying attention to.
