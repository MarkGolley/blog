Deploying My .NET Blog to Render

Recently, I set out to publish my personal .NET blog using Render, a platform I had not used before but which seemed well suited to simple hosting. It did not go entirely to plan. Here is what went wrong, what I learned, and how I eventually got it working.

My Goal

I had built a simple ASP.NET Core MVC blog project with a controller that serves posts from HTML files. I just wanted it hosted somewhere quickly, and Render seemed like a sensible option. It was — after a bit of trial and error.

Attempt 1: Native .NET – Static Site

Initially, I followed Render’s static site guide. The default build command I used looked like this:

dotnet publish MyBlog/MyBlog.csproj -c Release -o out

But this failed with:

bash: line 1: dotnet: command not found

Render describes their Static Site option as “great for blogs”, which sounded right at first. But it only works for purely static HTML/CSS/JS with no server-side code. That was the issue.

Attempt 2: Docker Deployment – Web Service

A Web Service supports server-side code and building from a Dockerfile, so it was a better fit for my C# ASP.NET Core application.

I wrote a Dockerfile using the official .NET 8 SDK image, published the app, and exposed port 80.

I also added a render.yaml file to tell Render how to build and deploy the application using Docker. Once this was in place, Render built and ran the app. But…

A Mysterious “Not Found”

Although the application deployed, when I visited the URL I was met with a “Not Found” page. It was not a 404 from my application, but Render’s own page.

After digging into it, I realised I had left in some non-standard code from a .NET 9 preview template I had previously experimented with. Once I removed it and returned to a straightforward middleware setup (UseStaticFiles(), MapControllerRoute(), etc.), everything worked as expected.

Success

After tidying up the Dockerfile and reverting my Program.cs to a simpler configuration, the site deployed correctly. I finally saw something other than “Not Found”.

In short: If you’re deploying a .NET application to Render, use a Web Service (not a Static Site), build with Docker, stick with .NET 8, and keep the middleware straightforward unless you’re certain you need something more involved.
