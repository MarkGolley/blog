AI Agents Can Do the Work, But Don't Skip the Learning

I use AI agents a lot now, both for this blog project and at work. They are genuinely useful. They save me time every week. But I have also noticed a pattern that feels important to say out loud: AI can help you ship faster while quietly reducing your understanding if you are not deliberate about how you use it.

The issue is not bad output every time. The issue is that the output is often good enough that you can click Proceed, merge, and move on before you have really understood what happened. This post is the practical approach I now use to avoid that trap.

What I See in Practice

From my own day-to-day use, this is the rough pattern:

  • Small tasks: usually very good. Boilerplate, formatting, repetitive transformations, and straightforward tests are often quick wins.
  • Medium tasks: often decent with supervision. The output is useful, but I still need to review assumptions and tighten the final shape.
  • Complex tasks: still weaker when there is heavy nuance, hidden context, and trade-offs.

So for me, AI is an accelerator, not a substitute for engineering judgement.

There is a second risk that sits next to learning: coding fluency. Even when I understand the design, heavy delegation can reduce how often I write core logic myself. Over time that can affect speed at the keyboard and confidence when AI is unavailable. I treat this as a separate discipline and something I still need to spend more deliberate time on.

The Temptation Is Real

Most of us have seen this flow:

  • The agent gives a plausible plan.
  • It writes code that compiles.
  • Tests pass.
  • You get an approval prompt.
  • You click Proceed and keep moving.

In the moment, that feels productive but over time it can become understanding debt. You are still the person expected to explain that change in code review, defend it in a production incident, and maintain it six months later. If you cannot explain it, you borrowed speed at the cost of confidence.

Why This Hits Newer Engineers Harder

If you are relatively early in your career, your main compounding advantage is how quickly you can build strong mental models. AI can help that but it can also block that, depending on how you use it.

If the agent becomes your memory, your debugger, your reviewer, and your architect all at once, your output can grow faster than your judgement. That gap is usually invisible until someone asks:

  • Why this approach over the alternative?
  • What is the failure mode if this is wrong?
  • What is the blast radius outside this diff?

You still need to answer those in your own words.

The Setup I Actually Use: Settings + Ledger

To keep speed and learning linked, I keep two files in my workflow:

  • Settings: a working agreement for how the agent and I collaborate.
  • Ledger: a running record the agent updates as I demonstrate concepts, so progress is tracked by evidence rather than memory.

1) Settings: make expectations explicit

My settings file enforces behaviour that keeps learning in the loop while still shipping:

  • Agent responsibility: explain recommendations in plain language first, then add precise terms so I can build vocabulary without losing the thread.
  • Before non-trivial changes: the agent must provide a short plan (3-5 lines), and I should confirm I understand the approach before code is written.
  • After changes: the agent must provide a short "what changed and why" summary, and I should be able to restate it in my own words.
  • Two understanding checks: one before coding (do I understand the plan?) and one before commit (can I explain what/why/failure mode/blast radius?).
  • Urgent override: keep one strict token (for me this is SHIP IT NOW) to skip checks when time-critical, but record that as understanding debt to revisit later.

This turns "I should probably understand this" into a concrete operating rule.

2) Ledger: track evidence, not confidence

The ledger tracks concepts by status with dates and evidence:

  • Learning: introduced and partially understood.
  • Proficient: explained correctly at least once.
  • Mastered: defended correctly multiple times.
  • Stale: previously known, recently shaky, needs re-teaching.

Promotions happen after demonstrated understanding. Wrong answers or "I do not understand" can demote a concept to Stale. Nothing gets silently deleted. That stops me mistaking familiarity for mastery.

In my workflow, the agent owns these updates by default, and I can still edit the ledger manually when needed.

One practical note: if your ledger starts empty, expect the agent to explain some basics early on. That settles down as entries build up, and you can manually edit or seed the ledger if you prefer a faster start.

The Day-to-Day Loop

This is the loop I try to stick to:

  1. Ask the agent for a plan and proposed change.
  2. Before coding, answer one focused comprehension question in my own words.
  3. After the change, explain what changed, why this route was chosen, and what could fail.
  4. The agent updates the ledger when a concept is genuinely demonstrated, and I can make manual adjustments if I need to correct or demote something.

It is still fast. The difference is that speed now compounds my capability instead of bypassing it.

What I Recommend If You Are Early Career

A simple delegation policy helps:

  • Green zone (delegate heavily): boilerplate, formatting, repetitive edits, straightforward test scaffolding.
  • Amber zone (delegate with checkpoints): feature logic, refactors, non-trivial tests.
  • Red zone (do not click Proceed blindly): security boundaries, data deletion, auth changes, incident fixes, and contract changes.
  • Black zone (manual-first): high-context platform code (for example Orleans-heavy distributed system internals) where hidden coupling is high and AI can disappear into plausible rabbit holes.

If you want a deeper security walkthrough, I cover this in Agent Guardrails in Practice: From Permission Popups to Policy Verification, including approval gates, policy checks, and audit logging.

In my experience, Black-zone work is where I can lose days fastest if I let the agent run too far without tight constraints. For Amber, Red, and Black work, force this check:

What does this change do?
Why this approach over the obvious alternative?
If this is wrong in production, what is the failure mode?
What else could this affect outside the diff?

If you cannot answer those, the blocker is not tooling. The blocker is understanding, and that is the next piece of work.

Key takeaway

If you are looping for more than an hour, stop and bring in a domain expert.

That handover point is easy to miss. A short conversation with someone deep in that domain can close key gaps quickly, introduce unfamiliar concepts properly, and save days of blind iteration.

A Starter Template You Can Copy

If you want to try this approach, start small:

Settings:
- Require a short plan before non-trivial code.
- Require a plain-English explanation after code changes.
- Require a pre-commit defence check.
- Keep one emergency override token for urgent shipping.

Ledger entry format:
- Concept
- Status (Learning / Proficient / Mastered / Stale)
- First seen date
- Last confirmed date
- One-line evidence

You do not need to replicate every rule on day one. Start with the minimum version you will actually use.

Copy-paste setup prompt Use this if you want an AI to scaffold your settings and learning ledger.

Copy this prompt and replace the bracketed placeholders:

You are helping me set up an AI learning-guardrail workflow for software engineering.

Context:
- Experience level: [junior / mid / senior]
- Main stack: [your stack]
- Current goals for the next 90 days: [goals]
- Team constraints: [code review expectations, branch policy, release pace]
- Risk tolerance for AI autonomy: [low / medium / high]

Task:
1) Ask me up to 7 focused questions before generating anything.
2) Generate two files:
   - settings.md
   - learning-ledger.md
3) settings.md must include:
   - plain-language-first explanations
   - a short pre-change plan requirement for non-trivial tasks
   - a post-change "what changed and why" requirement
   - one pre-change comprehension check
   - one pre-commit defence check
   - an emergency override token named SHIP IT NOW
4) learning-ledger.md must include:
   - headings: Mastered, Proficient, Learning, Parked, Stale
   - one-entry-per-concept format with first seen date, last confirmed date, and one-line evidence
   - promotion and demotion rules based on demonstrated understanding
5) Keep the workflow practical and lightweight so it can be used daily.
6) Add a 10-minute weekly review checklist.

Output format:
- Show settings.md in a code block.
- Show learning-ledger.md in a code block.
- Show a "Daily use" section in 5 bullets.

If the first draft feels heavy, ask for a "v2-lite" version with fewer rules but the same core checks.

Final Thought

I am not anti-AI. I am using it every day, and I am shipping faster because of it. I just do not want to confuse output velocity with actual growth. If my understanding does not keep pace, I have not really improved.

For me, explicit settings plus a learning ledger is the most practical way I have found so far to keep both moving in the right direction.
