Design Patterns Part 2

Repository

This one properly clicked for me. You create a repository interface that defines what you need (add, find, update), then a concrete class implements it. Your service takes the interface via DI so the implementation details are kept out of the way. The main benefit, as I see it, is that you can swap implementations later (different databases, in-memory for tests, and so on) without rewriting the service itself.

Strategy

This was an interesting one. You inject behaviour into a class instead of hard-coding it. The example I saw was a payment service that needs to handle different payment types. You define an interface such as Pay(), then create implementations (PayPal, Apple Pay, and so on). When you construct the service, you pass in the strategy you want. If you find yourself writing lots of if or switch branches to vary behaviour, a strategy is often a cleaner approach.

Kata Learnings

StringBuilder

This is a more efficient way to build strings because concatenation creates a new string each time. I had seen StringBuilder before but never really used it. I like that you can Append() repeatedly (and there are other useful methods such as Clear() and Replace()). For short strings I still tend to use +, but in loops or when building something larger this feels more appropriate.

Char vs String

I had not really thought about the distinction before: 'A' is a char and "A" is a string. In Python and JavaScript there is no distinction between single and double quotes, so this caught me out in C#. I am trying to be more deliberate about it now.
