Diagnosing Why It Was Inactive

When I first accessed the Pi-hole dashboard at:

http://192.168.1.XXX/admin

It loaded, but there was zero activity showing. Uh-oh. I realized there might be an issue with how the Pi-hole was set up, so I decided to investigate. Here's what I did and why:

1. Checking the Pi-hole Service

First, I checked if the Pi-hole service itself was running. If it wasn‚Äôt, that would explain the inactivity. I used:

sudo systemctl status pihole-FTL.service

This command shows the current status of the Pi-hole's DNS resolver. I was looking for "active (running)" in the output. If it's inactive, it might mean Pi-hole failed to start due to a configuration error.

2. Verifying the DNS Settings

Next, I checked my router to confirm it was correctly configured to forward DNS queries to the Pi-hole. If the router isn‚Äôt pointing to the Pi's address as the DNS server, the devices on the network won't send their queries to the Pi-hole.

Here‚Äôs why this is critical: without the router directing traffic to the Pi-hole, it can't filter or block anything. I logged into the router‚Äôs admin panel (usually found at 192.168.1.1) and set the Pi-hole's IP 192.168.1.XXX as the primary DNS server.

3. Testing DNS Resolution

To see if the Pi-hole was handling DNS queries, I ran a test from one of my devices. On my computer, I opened the terminal and used:

nslookup google.com

If the output didn‚Äôt show my Pi-hole‚Äôs IP address as the DNS server, it meant something was wrong with how DNS traffic was being routed. This test is crucial because it quickly determines if DNS requests are reaching the Pi-hole.

4. Checking the Raspberry Pi‚Äôs IP Address

Lastly, I confirmed that the Pi still had the correct IP address. Even though I assigned a static IP from the router, I used this command to verify:

hostname -I

If the Pi‚Äôs IP had changed, none of my DNS settings would work. That‚Äôs why assigning a static IP to the Pi earlier in the setup is so important‚Äîit prevents accidental misrouting.

Outcome

After these checks, I found that the router hadn‚Äôt saved the DNS configuration properly the first time. Reapplying the settings resolved the issue, and within moments, the Pi-hole dashboard started showing blocked ads!
